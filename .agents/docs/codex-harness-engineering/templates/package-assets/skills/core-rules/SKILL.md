# 核心开发规则

> 精简的硬性约束，确保代码质量和开发效率

---

## 1. 核心理念

1. **上下文第一**：垃圾进垃圾出，先理解再动手
2. **先结构后代码**：规划好框架再写实现
3. **一次只做一件事**：极致专注，小步迭代


## 2. 开发约束

### 架构与设计

4. 不得采用补丁式修改而忽视整体设计
5. 不得在业务代码中硬编码外部服务或混入 IO/网络/数据库副作用
6. 不得将状态提升到不必要的层级或跨请求共享可变状态

### 代码质量

7. 不得省略必要注释，代码必须对维护者可理解
8. 不得违反 SOLID 与 DRY 原则
9. 不得保留未使用的变量、函数或使用语义模糊的命名
10. 不得让单个函数或模块承担多个不相关职责

### 错误处理

11. 不得吞掉异常或使用空 catch 掩盖错误
12. 不得返回语义不清的错误结果（如混用 null/undefined/false）

### 开发流程

13. 不得臆测接口行为，必须先查询文档或源码
14. 不得在需求不清晰的情况下直接实现
15. 不得在缺乏上下文理解的情况下直接修改代码

---

## 3. 胶水编程约束

### 依赖使用

16. 不得自行实现底层或通用逻辑，必须优先复用成熟库
17. 不得复制依赖库代码到项目中修改或进行功能裁剪
18. 实际加载必须是完整生产级实现，不得使用简化版或 Mock

### 代码职责

19. 当前项目仅允许承担业务流程编排、模块组合、参数配置与适配职责
20. 不得在当前项目中重复实现依赖库已提供的功能
21. 所有导入模块必须在运行期真实参与执行，不得伪集成

---

## 4. 安全约束

22. 绝不引入、记录或提交密钥、API Key 或敏感信息
23. 确保所有用户输入都被正确验证和清理
24. 实施最小权限原则

---

## 5. 违反约束处理

当发现代码违反上述约束时：

1. 立即停止当前操作
2. 向用户说明违反了哪条约束
3. 提供符合约束的替代方案
4. 等待用户确认后再继续
