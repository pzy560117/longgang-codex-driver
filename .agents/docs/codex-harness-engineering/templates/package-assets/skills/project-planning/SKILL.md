# 项目计划工作流

> 将需求转化为层级化计划文档

---

## 工作流程

```
需求收集 → 深入分析 → 生成计划文档 → 审查确认
```

---

## 阶段 1：需求收集

### 必须澄清的问题

1. **项目目标**：核心功能？解决什么问题？
2. **功能模块**：分为哪几个主要模块？
3. **技术栈**：语言、框架、库的偏好？
4. **数据流向**：数据从哪来？到哪去？
5. **环境依赖**：外部服务、数据库、API？
6. **验收标准**：如何判断完成？
7. **约束条件**：时间、资源、技术限制？

### 需求总结格式

```markdown
## 目标
一句话描述要实现什么

## 非目标
明确不做什么

## 功能清单
1. [功能1]
2. [功能2]

## 技术约束
- [约束1]
- [约束2]
```

---

## 阶段 2：计划文档结构

### 文件命名规范

- 格式：`plan_XX_任务名.md`
- 编号：从 01 开始连续递增
- plan_01 必须是"总体计划"（1级）

### 层级结构

```
plan/
├── plan_01_总体计划.md      # 1级
├── plan_02_[模块名].md      # 2级
├── plan_03_[子任务名].md    # 3级
├── plan_04_[子任务名].md    # 3级
├── plan_05_[模块名].md      # 2级
└── ...
```

### 层级说明

| 层级 | 内容 | 详细程度 |
|:---|:---|:---|
| 1级 | 总体计划 | 概要 |
| 2级 | 模块计划 | 适中 |
| 3级 | 具体任务 | 详细 |

---

## 阶段 3：可视化要求

每个层级需包含匹配的可视化：

- **1级**：系统逻辑图、模块关系矩阵、里程碑时间线
- **2级**：模块流程图、接口协作图、资源分配表
- **3级**：任务流程图、风险监控表

---

## 计划文档 YAML 头

```yaml
---
level: 1/2/3              # 层级
file_id: plan_XX          # 文件编号
parent: plan_XX           # 父任务编号
children: [plan_XX, ...]  # 子任务编号
status: pending           # 状态
created: YYYY-MM-DD HH:mm # 创建时间
estimated_time: XX分钟     # 预估耗时
---
```

---

## 关键原则

### 必须遵守

1. 只生成计划，不编写实际代码
2. 使用抽象描述和占位符
3. 确保计划文档信息完整可执行
4. 严格遵循层级结构
5. 连续编号不跳号

### 禁止行为

1. 不要编写实际代码
2. 不要使用具体的文件名示例
3. 不要使用具体的函数名示例
4. 只生成 plan_XX.md 文件
