# 独立摄影师预订系统专项指导

## 重点论证的问题

优先围绕以下问题组织正文：

1. 三类角色的认证与授权边界
2. 摄影师档期冲突控制
3. 订单从创建到支付、签收、评价、退款的状态闭环
4. 前后端围绕档期和订单状态的联动机制
5. 管理员审核与退款仲裁的治理能力

## 应谨慎处理的内容

以下内容若实际未完整落地，不得写成已实现能力：

1. 微服务
2. Nacos / Gateway / Seata / OpenFeign
3. 真实支付平台接入
4. MinIO / OSS 对象存储
5. 选片、通知、推荐等扩展功能

正确写法：

- “开题阶段曾考虑……”
- “经权衡后最终采用……”
- “当前版本以……替代，用于验证……”
- “该能力保留为后续扩展方向”

## 最容易写歪的地方

1. 把第 3、4 章写成“后端实现 + 前端实现”
2. 把页面截图堆成产品展示手册
3. 把订单、档期、权限这些真正有技术含量的问题写轻了
4. 把开题设想和最终实现混在一起
5. 把构建通过写成测试结论

## 第5章实现建议

如果学校模板要求单独写“系统实现”章，当前项目更稳妥的写法不是回避前后端，而是把它们放到第5章中集中展示。

推荐写法一：

1. `5.1 客户端主要页面实现`
2. `5.2 摄影师端页面实现`
3. `5.3 管理端页面实现`
4. `5.4 后端关键业务实现`

推荐写法二：

1. `5.1 前端主要页面实现`
2. `5.2 后端关键模块实现`

无论采用哪种写法，都要满足：

1. 前端部分必须放**真实浏览器运行截图**，不能放原型图或静态设计图。
2. 截图优先覆盖：首页/搜索页、摄影师详情与档期页、客户订单页、摄影师工作台、管理员审核或退款治理页。
3. 每张截图后都要解释“页面做了什么、和哪类状态联动、为什么能证明实现成立”。
4. 后端部分优先放真正有论证价值的代码，例如：
   - `OrderService`
   - `AvailabilityService`
   - `PaymentService`
   - `SecurityConfig`
   - `RedisDistributedLock`
5. 代码片段要短，只截关键逻辑，且必须配文字解释它解决了什么问题。

## 图表专项提醒

至少要保证以下图表真正落地：

1. 总体架构图
2. 角色或业务主流程图
3. E-R 图
4. 订单状态图
5. 预约支付时序图
6. 首页或搜索页截图
7. 摄影师详情与档期截图
8. 客户订单页截图
9. 摄影师工作台截图
10. 管理员审核或治理截图

若某些图文件已生成但正文未使用，应二选一：

- 正式插入并解释
- 删除不用，避免图号体系混乱
