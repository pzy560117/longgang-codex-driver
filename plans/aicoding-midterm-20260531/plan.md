# Plan：AICoding 中期评审收口

## 1. 项目判断

当前项目已经具备 AICoding 工程化试点的基础，不需要重新定义方向。接下来要做的是把已有需求、实现、测试、trace 和评审口径收口成一个能被项目组、技术负责人和客户方理解的阶段成果。

核心表达：

> 以统一导出平台为样板，验证 AICoding 如何在 Harness 控制下进入企业级软件工程过程，形成可管理、可测试、可追踪、可复盘、可交接的开发闭环。

## 2. 阶段目标

### 2.1 必达目标

| 目标 | 完成定义 | 对应材料 |
| --- | --- | --- |
| 方法论可讲清 | 能说明 Harness 如何约束 AI Coding，从需求进入到验证提交 | `docs/harness/*`、本目录、评审材料 |
| 样板工程可演示 | 统一导出平台最小闭环可在本地受控环境演示 | `npm run demo:local`、`npm run demo:local:smoke`、测试报告 |
| 证据链可追溯 | 能找到 PRD、架构约束、任务、测试、trace、报告 | `docs/product/`、`docs/architecture/`、`plans/`、`docs/testing/`、`traces/` |
| 边界可解释 | 明确本地 Docker/mock 已验证，外部生产环境未声明完成 | `docs/testing/full-acceptance-test-report.md` |
| 下一阶段可执行 | 能拆出团队化、环境迁移、真实依赖接入和知识回流任务 | `task-breakdown.md` |

### 2.2 不在本阶段承诺

- 不承诺外部生产 MySQL 已接入。
- 不承诺 live OSS/S3 已完成生产验收。
- 不承诺客户真实业务数据源已完成联调。
- 不承诺该服务已经可直接生产上线。
- 不把能源优化线纳入本计划的主交付范围。

## 3. 评审成果组合

| 成果 | 内容 | 展示方式 |
| --- | --- | --- |
| AICoding 工程化方法论 | Harness 流程、角色介入点、质量门禁、trace 证据 | 流程图 + 仓库路径 |

| 下一阶段实施计划 | 团队级流程、真实环境接入、可观测性、知识沉淀 | 任务拆解 |

## 5. 风险

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 只展示代码，忽略 Harness 过程 | 被理解为普通工具试用 | 演示必须从需求、任务、约束、测试、trace 开始 |
| 把本地 mock 证据说成生产可用 | 评审后落地风险高 | 明确证据边界，本阶段是 docker/mock 受控验收 |
| 功能范围继续扩张 | 5 月底无法收口 | 不新增非必要功能，优先演示闭环和证据 |
| 真实客户环境未接入 | 无法证明生产集成 | 作为下一阶段真实依赖接入任务，不伪造完成 |
| 团队看不到过程材料 | 难以支撑企业级方案 | 所有关键材料必须落到仓库路径或团队可访问位置 |
