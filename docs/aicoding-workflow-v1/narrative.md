# AICoding 流程讲述脚本

本文用于把当前项目讲给领导、技术负责人和团队成员听。重点不是逐条解释内部脚本，而是让听众理解：为什么需要这套流程、它解决什么问题、现有项目已经证明了什么、还需要一起优化什么。

## 开场口径

可以这样开场：

> 这次我们不是单纯展示 AI 写了一个导出功能，而是用导出功能做样板，验证 AI Coding 能不能进入企业级研发流程。企业真正关心的不是 AI 写了多少代码，而是需求能不能被正确输入，执行过程能不能被团队看到，质量能不能被技术负责人验收，结果能不能被复盘，下一类业务功能能不能复制这套方法。

不建议这样讲：

- “AI 已经把功能开发完了。”
- “这个工具可以自动替代开发。”
- “本地能跑，所以流程已经完成。”

原因：这些说法容易把讨论带偏到工具能力或个人演示，反而弱化企业级研发最关心的过程、质量和责任边界。

## 第一段：先讲原问题

核心观点：

> AICoding 的关键问题不是代码生成，而是过程不可见。

可以展开为：

- 需求、PRD、架构约束、任务拆分、执行轨迹、测试证据、review 结论和提交记录，如果只散落在个人电脑、私人账号或聊天上下文里，团队无法共同判断质量。
- 技术负责人如果只能看到最终代码，看不到中间决策，就很难判断“为什么这样做”和“有没有绕过关键门禁”。
- 产品和业务专家如果不能在需求、样例、异常流阶段介入，AI 很容易沿着错误理解继续执行。
- 测试和运维如果只在最后看到结果，环境边界、失败态和真实依赖接入会被推迟暴露。

过渡句：

> 所以我们这次不是只做一个功能，而是把 AI 参与研发的过程固定下来，让它可以被团队查看、讨论和改进。

## 第二段：再讲当前项目已经具备什么

可以用“六件事”来讲，不需要一开始展开所有文件细节：

1. **需求输入有位置**
   - 产品文档、验收标准和状态矩阵在 `docs/product/`。
   - 后续 AI 执行不依赖临时口头描述。

2. **架构约束有位置**
   - 架构边界在 `docs/architecture/constraints.md`。
   - 明确交付形态、API 边界、数据库边界、worker 边界、测试替身策略和禁止实现。

3. **任务拆分有位置**
   - `task.json` 是真实任务队列。
   - 每个任务有依赖、owned paths、验证命令和验收条件。

4. **执行过程有机制**
   - `codex-loop.ps1` 按 single task 模式串行执行。
   - 每轮只处理一个依赖满足且未通过的任务，避免聊天式随意推进。

5. **质量验证有证据**
   - 验证矩阵、测试报告、review 文档和 trace 共同构成证据链。
   - 不能只凭“功能能跑”声明企业级完成。

6. **结果沉淀有边界**
   - commit、trace、progress、review 和知识沉淀用于复盘。
   - 本地/mock/Docker/live 证据必须分开写清楚。

## 第三段：讲导出功能为什么适合作为样板

核心观点：

> 导出功能不是最终目标，而是一个复杂度足够高的验证样板。

它覆盖了真实工程中容易暴露问题的边界：

- HTTP API 和 OpenAPI 契约。
- MySQL schema、migration 和 repository。
- 异步任务、worker、锁租约、重试、取消和接管。
- 文件生成、对象存储边界和下载权限。
- 认证、权限、参数校验和公开错误脱敏。
- 审计日志、任务状态、验收测试和 review。

可以这样总结：

> 如果这类功能都能被拆成可验证任务，并且每一步都能留下证据，那么这套流程才有资格被复制到其他业务功能。

## 第四段：讲现在仍然要讨论什么

不要把现状讲成完美闭环。更好的口径是：

> v1 已经有底座，但它还需要团队一起优化。我们现在要讨论的不是要不要 AI Coding，而是怎么让它变成团队可接受、可监督、可复用的研发方式。

建议重点抛出这些问题：

- 产品需求给 AI 的格式是否足够清楚？
- 架构约束是否能防止 AI 走捷径？
- 任务粒度是否适合真实团队协作？
- 技术负责人最关心的验收证据是否容易找到？
- 测试证据是否能复现，是否区分 mock、本地、Docker 和真实环境？
- review 应该在哪些阶段卡住，哪些阶段只做提示？
- 哪些过程证据需要放到团队都能访问的位置？
- 下一个业务功能如何复用这套流程？

## 结尾口径

可以这样收尾：

> 这套流程的价值不是让 AI 独立完成所有事情，而是把 AI 的执行能力放进可控的研发链路里。人负责目标、约束、评审和验收，AI 负责在明确边界内执行、补证据、修问题。后续我们要一起优化的是协作方式和质量门禁，而不是单纯比较哪个工具生成代码更快。

## 常见质疑与回应

| 质疑 | 回应口径 |
| --- | --- |
| 这不就是自动化脚本吗？ | 脚本只是执行层，核心是需求、任务、约束、验证、review 和证据链被固定下来。 |
| AI 写的代码可信吗？ | 不靠“相信 AI”，而靠架构约束、测试命令、review gate 和可复现证据来判断。 |
| 会不会流程太重？ | v1 先针对企业级复杂功能，后续可以按风险分级裁剪轻量版。 |
| 本地验证能代表生产吗？ | 不能，所以文档必须区分本地、Docker/mock 和 live evidence，不能混写。 |
| 谁来负责最终质量？ | 技术负责人仍负责验收，AI 只是执行者；流程的目的就是让负责人有证据可审。 |
| 后续如何推广？ | 先选一个中等复杂度新功能复用流程，再根据问题调整模板和门禁。 |
