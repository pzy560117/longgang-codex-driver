# AICoding 流程共创讨论会方案

本文用于组织一次讨论会，让团队基于当前项目已有流程提出优化建议。会议目标不是重新设计 Harness，也不是争论某个 AI 工具好坏，而是识别 v1 流程进入团队研发时的缺口。

## 会议目标

会后应形成三类输出：

1. 当前流程中已经可以复用的部分。
2. 进入团队协作前必须补齐的问题。
3. 下一轮可执行的优化 backlog。

## 参会角色

| 角色 | 关注重点 |
| --- | --- |
| 产品负责人 | 需求、PRD、验收标准是否适合输入给 AI |
| 业务专家 | 业务规则、异常流、样例数据是否表达清楚 |
| 技术负责人 | 架构约束、实现边界、review 和质量门禁是否可信 |
| 开发人员 | 任务粒度、上下文输入、代码接手成本是否合理 |
| 测试人员 | 测试矩阵、验收证据、失败态覆盖是否可复现 |
| 运维或平台人员 | 环境、部署、权限、日志和真实依赖边界是否清楚 |
| 项目负责人 | 流程成本、节奏、产出可见性和推广路径 |

## 会前材料

建议提前发送以下路径或截图，不要求所有人读完整仓库：

- `docs/product/aicoding-workflow-v1-plan.md`
- `docs/aicoding-workflow-v1/README.md`
- `docs/aicoding-workflow-v1/narrative.md`
- `task.json`
- `docs/testing/verify-matrix.md`
- `docs/testing/full-acceptance-test-report.md`
- `docs/reviews/requirements-complete-review.md`

会前提醒：

> 本次讨论请重点看流程是否透明、可控、可验收，不需要逐行审查代码。

## 会议流程：60 分钟

### 1. 背景对齐（10 分钟）

主持人说明：

- 为什么只展示最终代码不够。
- 为什么导出功能适合作为样板。
- 当前 v1 已有流程底座，不从零讨论。

要避免：

- 一开始讨论具体工具优劣。
- 一开始陷入某个代码细节。
- 把“本地能跑”当成完整验收。

### 2. 样板链路 walkthrough（15 分钟）

按下面顺序展示：

1. 需求来源：`docs/product/`
2. 架构约束：`docs/architecture/constraints.md`
3. 任务队列：`task.json`
4. 执行机制：`codex-loop.ps1`
5. 验证矩阵：`docs/testing/verify-matrix.md`
6. 验收报告：`docs/testing/full-acceptance-test-report.md`
7. review：`docs/reviews/requirements-complete-review.md`
8. trace / commit：按当前仓库实际记录展示

讲述重点：

- 每一步都要回答“谁能看、看什么、用来判断什么”。
- 如果某项证据当前失败或不稳定，要如实标注为优化项。

### 3. 分角色提问（20 分钟）

主持人逐类收集，不要让讨论散成泛泛意见。

#### 产品视角

- PRD 里哪些内容 AI 最容易误解？
- 验收标准应该写到什么粒度？
- 是否需要固定需求模板？
- 业务样例和异常样例应该由谁提供？

#### 技术视角

- 架构约束是否足以阻止错误实现？
- 哪些地方必须人工 review？
- task 粒度是太大、太小，还是刚好？
- 哪些证据技术负责人现在还不信？

#### 测试视角

- 当前验证矩阵是否能追踪到每个核心需求？
- 失败态是否覆盖足够？
- 测试报告是否能复现？
- mock、本地、Docker、live evidence 是否区分清楚？

#### 协作视角

- 过程记录是否放在团队都能访问的位置？
- 谁在什么时候介入最合适？
- 如果 AI 卡住，应该如何升级给人？
- 哪些信息不应该留在私人账号或聊天上下文里？

### 4. 收口与排序（10 分钟）

把意见分成三类：

| 优先级 | 定义 | 示例 |
| --- | --- | --- |
| P0 | 不解决就不能对外声称企业级可用 | 验收报告失败、证据不可复现、真实依赖边界混乱 |
| P1 | 不解决会影响团队规模化协作 | 需求模板不稳定、任务粒度不一致、review 入口不清楚 |
| P2 | 改善体验和推广效率 | 汇报图不够清晰、路径索引不够友好、模板说明不够顺 |

每个问题至少记录：

- 问题描述。
- 影响角色。
- 当前证据或例子。
- 建议 owner。
- 验证方式。

### 5. 下一步确认（5 分钟）

主持人确认：

- 第一批 P0/P1 优化项。
- 下一个用于复用流程的业务功能。
- 下一次复盘时间。

## 主持人注意事项

- 讨论要围绕“流程是否可验收、可复盘、可复制”。
- 对工具能力的争论先记录，不在会上展开。
- 对失败证据不要掩盖，直接转成 P0/P1 优化项。
- 不要让“AI 自动化程度”压过“人在哪里介入和负责”。
- 每条改进意见最后都要落到 owner 和验证方式，否则只是观点。
