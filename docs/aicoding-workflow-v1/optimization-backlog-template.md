# AICoding 流程优化 Backlog 模板

本文用于把讨论会意见整理成后续可执行任务。它不是正式 `task.json`，而是进入 `task.json` 前的人工整理层。

## 使用方式

1. 会后先把所有意见填入“候选池”。
2. 按 P0/P1/P2 排序。
3. 每条只保留一个清晰问题，不把多个问题塞进一个条目。
4. 对 P0/P1 补齐证据路径和验证方式。
5. 再决定哪些转成正式 `task.json` 任务。

## 优先级定义

### P0：可信交付阻塞

满足任一条件即可列为 P0：

- 会导致流程无法被技术负责人验收。
- 会导致未验证内容被误报为完成。
- 会导致 mock、本地、Docker、live evidence 混写。
- 会导致敏感信息泄露到文档、trace 或仓库。
- 会导致团队无法复盘关键决策。

示例：

- 全量验收报告显示 FAIL，但对外口径仍写成 PASS。
- 关键测试命令不可复现。
- 真实依赖未接入，却被写成 live evidence。

### P1：规模化协作阻塞

满足任一条件可列为 P1：

- 团队成员能看懂结果，但难以参与过程。
- 需求、架构、测试或 review 模板不稳定。
- 任务粒度导致 AI 执行容易跑偏或人类 review 成本过高。
- 缺少角色介入点，导致问题发现太晚。

示例：

- PRD 没有固定验收示例格式。
- 任务拆分缺少 owner 或 owned paths。
- review 结论难以对应到具体需求 ID。

### P2：体验和推广优化

满足任一条件可列为 P2：

- 不影响可信交付，但影响汇报效率。
- 不影响团队执行，但影响新人理解。
- 不影响质量门禁，但可以减少沟通成本。

示例：

- 缺少一页式流程总览图。
- 证据地图路径太分散。
- 汇报材料没有按听众角色区分版本。

## 转入正式任务前检查

转成 `task.json` 前逐项确认：

- 是否有明确 truth source。
- 是否有清晰 owned paths。
- 是否有可执行验证命令。
- 是否会改动现有导出平台结论。
- 是否需要先处理工作区未提交或失败证据。
- 是否涉及真实外部依赖、账号、token 或生产数据。

如果答案不清楚，先保持在本 backlog，不要急着进入 driver。
