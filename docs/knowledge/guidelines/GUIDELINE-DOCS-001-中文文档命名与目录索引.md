# 中文文档命名与目录索引

## 背景

近期交接文档大量使用英文文件名，接手人员需要逐个打开才知道内容。后续新增面向人读的项目文档时，应优先让文件名和目录索引直接表达用途。

## 规则

- 面向业务、交接、运维、测试报告、经验总结的 Markdown 文档，文件名优先使用中文，做到短、准、能看出内容。
- 技术约定、脚本入口、标准文件、工具强依赖文件可保留英文，例如 `README.md`、`AGENTS.md`、`task.json`、OpenAPI、配置模板和已有工具约定路径。
- 新增或批量调整文档时，同步维护一个目录索引，说明每个目录放什么、关键文件解决什么问题、谁应该先读。
- 索引只写定位信息，不复制正文；每行说明控制在一句话内。
- 如果不能重命名已有英文文档，至少在索引中提供中文标题和用途说明。

## 推荐做法

新增文档前先判断读者：

| 读者 | 命名建议 |
| --- | --- |
| 业务 / 产品 / 接手研发 / 运维 | 中文短名，例如 `生产接入交接指南.md` |
| 自动化工具 / 约定入口 / 第三方标准 | 保留英文或标准名 |
| 历史已引用文件 | 不强行改名，先补索引 |

## 替代方案

如果需要兼容已有英文路径，保留原文件，同时在 `docs/文档索引.md` 里给出中文说明。后续有明确迁移窗口时，再集中重命名并更新引用。

## 适用阶段

- 文档交付
- 运维交接
- 测试证据归档
- 知识沉淀
- 新人接手
